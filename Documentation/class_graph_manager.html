Detailed Description

Manages all graphs in Construct.

Graphs are the primary data storage class for information connecting more than one node. Ex. which agents can interact with which other agent, which agent has which knowledge, which knowledge can be used on a communication medium. The graph manager loads all graphs from the input XML file submitted to Construct during construction. This included adding the specified links and applying any generators in the associated XML element. Generators can not be applied to existing graphs after the XML loading has taken place.

The graph manager offers three styles for loading graphs, a hard load, a soft load, and a bare-bones check. hard_load will either set the submitted pointer to the pointer of the request graph or cause a premature exit of the program. This can happen if the requested cannot be found, or the submitted parameters do not match the requested graph's parameters. soft_load performs similar checks, but if the requested graph is not found, a null pointer is returned. get_network does not perform any checks, does not cast the pointer from its parent class, and if the requested graph is not found, a null pointer is returned.

Graphs can not be constructed except by this class. Graphs can be added via add_network and add_3dnetwork. If there already exists a graph with the submitted name a routine similar to soft_load is performed. If no graph exists with that name, a new graph is constructed with the submitted parameters.

Complexity

When a new Graph is created, complexity depends on whether the dimensions are dense or sparse. If a dimension is dense, a vector is initialized for that dimension and has a complexity of O(n) where n is the nodeset size. If the Graph already exists, the complexity is constant.

Sets the submitted pointer to the requested 2d graph.

The only identifier required to find a graph is its name. The remainder of the parameters are required to ensure that the graph matches the expected parameters. A mismatch in nodeset can cause the access of an element outside the graph's bounds. Additionally the edge data types must match in order for the parent pointer to be cast into the correct Graph pointer.

When a graph is successfully found and its parameters are matched with input, the name of the accessing entity is entered into an access table. This allows users to examine which entities have accessed a graph and more importantly which entities can modify it. These accesses can be viewed using #get_accesses. This method is only available when the DEBUG compilation flag is active.

Parameters
nameName of the graph being requested.
srcThe source nodeset that the requested graph must match.
trgThe target nodeset that the requested graph must match.
gA constant reference that gets set to the requested graph pointer if the graph is found.
model_nameName of the accessing entity.

Example

int main()
{
const Graph<bool>* knowledge;
construct->getGraphManager()->hard_load(
graph_names::knowledge,
construct->getNodesetManager()->get_nodeset(nodeset_names::agent),
construct->getNodesetManager()->get_nodeset(nodeset_names::knowledge),
ktm, "example model");
std::cout << "graph name: " << ktm->name << std::endl;
}

Output: 

graph name: knowledge network

Complexity

Constant.

Exception Safety

If either source nodeset or target nodeset do not match up between the requested graph and input, a premature exit of the program will occur.
